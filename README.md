# Studio Fujisaki

Free **Japandi wallpapers** made from public-domain Japanese art.
A static site ‚Äî plain **HTML + CSS + vanilla JS**, no build step. Push to GitHub, it's live.

- Quiet, minimal, light Japandi design (a Dark Japandi variant can be added later via CSS variables).
- All artwork data lives in one file: [`data/wallpapers.json`](data/wallpapers.json).
- Cards and pages are generated by JS from that JSON.
- Thumbnails (lightweight WebP) live in the repo; **full-size images live in external storage (Cloudflare R2)**, never in git.

**Paths are relative on purpose.** Internal links, CSS/JS, and thumbnails never use a

URLs are the full-size image links (`pcUrl`/`spUrl`), which point to R2.

## Setup

No build tools are needed to run the site ‚Äî it's static. To preview locally, serve the
folder over HTTP (the JSON is loaded via `fetch`, so `file://` won't work):

```bash
# any static server works; e.g. with Node installed:
npx serve .
# then open the printed http://localhost:... URL
```

To use the helper scripts, install their dependencies once:

```bash
cd scripts
npm install
```

## `config.js` ‚Äî what to replace

Everything site-wide is defined once in [`config.js`](config.js). Replace the placeholders:

| Constant        | Replace with                                                        |
|-----------------|---------------------------------------------------------------------|
| `BRAND_NAME`    | Your brand (default `Studio Fujisaki`)                              |
| `TAGLINE`       | Your tagline                                                        |
| `R2_BASE_URL`   | Public base URL of your R2 bucket (must match `.env`'s `R2_PUBLIC_BASE_URL`) |
| `KOFI_URL`      | Your Ko-fi page                                                     |
| `SITE_URL`      | Your public site URL (used for OGP / sitemap absolute URLs)         |
| `COLLECTIONS`   | Collection slugs / display names / descriptions                    |

Also replace `https://REPLACE-ME.example.com` in **`sitemap.xml`** and **`robots.txt`**
(or regenerate the sitemap with `npm run gen-sitemap -- --base https://your-domain.com`).

## `.env` ‚Äî credentials (never commit)

R2 credentials are **never** hard-coded. Copy the template and fill it in:

```bash
cp .env.example .env      # .env is git-ignored ‚Äî keep it that way
```

`.env` holds:

| Variable               | Meaning                                                    |
|------------------------|------------------------------------------------------------|
| `R2_ACCOUNT_ID`        | Cloudflare account id                                      |
| `R2_ACCESS_KEY_ID`     | R2 access key                                              |
| `R2_SECRET_ACCESS_KEY` | R2 secret key                                              |
| `R2_BUCKET`            | Bucket name (e.g. `studio-fujisaki`)                       |
| `R2_PUBLIC_BASE_URL`   | Public base URL serving the bucket (matches `config.js`)   |
| `R2_ENDPOINT`          | S3 API endpoint, `https://<account_id>.r2.cloudflarestorage.com` |

> ‚öÝÔ∏è **Never commit `.env`.** It is excluded by `.gitignore`. The repo is public;
> committing credentials would expose your storage. The `upload-r2` script reads these
> values **only** from the environment.

## GitHub Pages ‚Äî publishing

1. Create a **public** GitHub repository and push this folder to it.
2. Repo **Settings ‚Üí Pages** ‚Üí Source: **Deploy from a branch** ‚Üí Branch `main`, folder `/ (root)` ‚Üí Save.
3. Wait for the green check; your site is at `https://<user>.github.io/<repo>/`.

Because all internal paths are relative, the project-subpath URL works with no changes.

## Adding a new wallpaper (step by step)

Full-size images stay out of git; only the WebP thumbnail is committed.

```
1. Put the full-size images in a working folder (git-ignored), named exactly:
      originals/<slug>/pc.jpg     (desktop / 4K)
      originals/<slug>/sp.jpg     (phone / portrait)

2. Generate the thumbnail:
      cd scripts
      node gen-thumb.js ../originals/<slug>/pc.jpg <slug>
   ‚Üí writes  thumbs/<slug>.webp

3. Upload full-size images to R2 (reads credentials from .env):
      node upload-r2.js --slug <slug> --dir ../originals/<slug>
   ‚Üí uploads pc.jpg / sp.jpg and writes pcUrl/spUrl into wallpapers.json

4. Add the metadata block (auto-slug, validates JSON + required fields + thumb exists):
      node add-wallpaper.js --title "‚Ä¶" --artist "‚Ä¶" --era "‚Ä¶" \
                            --museum "‚Ä¶" --collection <collection-slug>
   (or run with no flags for an interactive prompt)

   Tip: run add-wallpaper BEFORE upload-r2 if you prefer; upload-r2 will then fill
   the real URLs into the entry it finds. Either order works.

5. (optional) Regenerate the sitemap:
      node gen-sitemap.js --base https://your-domain.com

6. Commit & push:
      git add data/wallpapers.json thumbs/<slug>.webp sitemap.xml
      git commit -m "Add <slug>"
      git push
```

### The scripts

| Script            | What it does                                                                 |
|-------------------|------------------------------------------------------------------------------|
| `gen-thumb`       | Full image ‚Üí 600px WebP thumbnail in `/thumbs` (uses `sharp`).               |
| `add-wallpaper`   | Appends one validated block to `wallpapers.json` (auto slug; checks JSON, required fields, thumb existence, unique slug). Flags or interactive. |
| `upload-r2`       | Uploads `pc.jpg`/`sp.jpg` to R2 (S3-compatible, `@aws-sdk/client-s3`) and writes `pcUrl`/`spUrl` back into the JSON. Credentials from `.env` only. |
| `gen-sitemap`     | (bonus) Regenerates `sitemap.xml` from the JSON.                             |

`wallpapers.json` schema (per artwork):

```json
{
  "slug": "‚Ä¶", "title": "‚Ä¶", "artist": "‚Ä¶", "era": "‚Ä¶", "museum": "‚Ä¶",
  "collection": "ink-and-mist | sage-and-stone | warm-sand",
  "thumb": "thumbs/<slug>.webp",
  "pcUrl": "https://<r2-base>/<slug>/pc.jpg",
  "spUrl": "https://<r2-base>/<slug>/sp.jpg"
}
```

(No tablet field. Add new keys freely ‚Äî the renderer ignores unknown ones.)

## Moving to a custom domain

1. Add your domain in **Settings ‚Üí Pages ‚Üí Custom domain**, and create a `CNAME` file
   (GitHub Pages does this for you) pointing to your domain.
2. Configure DNS at your registrar (a `CNAME` to `<user>.github.io`, or the four A records
   for an apex domain ‚Äî see GitHub's docs).
3. Update `SITE_URL` in `config.js`, and the host in `robots.txt` / `sitemap.xml`
   (or `npm run gen-sitemap -- --base https://your-domain.com`).
4. Nothing else needs to change: internal links are relative, so they keep working when the
   base moves from a subpath to the domain root.
